# MCUQuickStart — 开发记录

## 项目目标

桌面 GUI 工具，选择芯片型号后一键生成 Keil5 打开即编译的嵌入式工程模板。

## 技术栈

- Python 3 + PyQt6（GUI）
- JSON 驱动（芯片数据库）
- 模板 + 占位符替换（代码和工程模板）
- 官方 SDK 文件实时拷贝（不打包库文件）

## 开发历程

### 第一阶段：核心架构（2026-05-19 下午）

**1. 项目骨架**
- 目录结构：`src/core/`（逻辑）+ `src/gui/`（界面）+ `src/resources/`（数据+模板）
- `requirements.txt`：PyQt6

**2. 芯片数据库（chip_db.py）**
- JSON 定义每款芯片参数（RAM/ROM、启动文件、flash 驱动、预处理器宏）
- `ChipDatabase` 类：加载、查询、deep copy 保护内部状态

**3. 模板引擎（template_engine.py）**
- `render()`：`{{PLACEHOLDER}}` 替换
- `render_directory()`：递归目录渲染
- 支持 UTF-8 + GBK 编码自动识别（中文注释兼容）

**4. SDK 管理器（sdk_manager.py）**
- 路径配置持久化到 `~/.mcu_template_config.json`
- `find_file()` / `_find_dir()`：多前缀尝试（`""` → `"Firmware/"` → `"Firmware/CMSIS/"`）
- `copy_firmware()`：从 SDK 拷贝 CMSIS 核心头文件、设备头文件、系统源文件、启动文件、标准外设库

**5. 工程生成器（project_generator.py）**
- 编排完整生成流程：拷贝固件 → 建目录 → 渲染模板 → 生成 uvprojx
- `_build_uvprojx_vars()`：构建 Keil 工程变量映射
- `_code_vars()`：构建代码模板变量映射
- `_copy_sdk_templates()`：从 SDK Template 目录实时拷贝芯片专属文件
- `_kb_to_hex()`：内存大小换算

### 第二阶段：GUI + 模板（2026-05-19 下午）

**6. PyQt6 主窗口（main_window.py）**
- SDK Root 路径配置
- 芯片系列/型号级联下拉
- 代码模板选择（空壳 / LED / UART）
- 项目名 + 输出目录
- 生成按钮 + 日志输出

**7. 代码模板**
- 工具自带：`delay.c/h`、`sysconfig.c/h`、`debug_print.h`、`retarget_printf.c`、`RTE_Components.h`
- SDK 实时拷贝：`gd32f10x_it.c/h`、`gd32f10x_libopt.h`、`systick.c/h`、`main.h`
- main.c 变体：empty（空循环）、LED（GPIO 闪烁）、UART（printf 输出）

**8. Keil5 uvprojx 模板**
- 从参考工程提取，变量化：`{{PROJECT_NAME}}`、`{{CHIP}}`、`{{STARTUP_FILE}}`、`{{FLASH_DRIVER}}`、`{{DEVICE_DEFINE}}` 等
- 内存映射（IRAM/IROM）、CPU 类型、Pack ID 全部占位符化

### 第三阶段：集成测试 + Bug 修复（2026-05-19）

**Bug 修复清单：**

1. **编码问题**：`delay.c` 含 GBK 中文注释 → template_engine 加多编码回退
2. **Vendor 不匹配**：GUI 用 "GD32" 存 SDK 路径，但 chip JSON vendor 是 "GigaDevice" → 引入 `sdk_key` 字段独立于 Keil vendor
3. **启动文件 IAR 覆盖**：rglob 同时匹配 ARM 和 IAR 同名 `.s` 文件 → 加 `/IAR/` 路径过滤
4. **设备头文件缺失**：`gd32f10x.h` 未拷贝到生成项目 → `copy_firmware` 增加 device include 拷贝步骤
5. **Keil DFP 设备名**：`GD32F103C8T6` 应写为 `GD32F103C8`（无封装后缀）→ 芯片 JSON 增加 `device` 字段
6. **密度宏写死**：`GD32F10X_MD` 家族级写死，HD 芯片编译不过 → 改为 per-chip 的 `define` 字段
7. **`_find_dir` rglob 错误匹配**：找到了 lwip 的 Include 而非设备头文件目录 → 加 `Firmware/CMSIS/` 前缀尝试 + rglob 优先 Firmware 路径

### 第四阶段：F4 系列支持（2026-05-19 晚）

**1. GD32F4xx 芯片数据库**
- 9 款芯片：F405×1、F407×4、F450×4
- 每芯片独立 `define`、`flash_driver`、`startup`、`device`

**2. F4 uvprojx 模板**
- 从 F4 SDK Keil5 模板生成
- Cortex-M4 + FPU2 + IRAM2（CCM RAM）
- DLL 参数：`-pCM4` + `-MPU`

**3. F4 代码模板**
- GPIO API 不同于 F10x：`gpio_mode_set()` + `gpio_output_options_set()` 替代 `gpio_init()`
- UART 需要 AF 模式：`gpio_af_set()` 配置复用功能
- 延时函数：`delay_1ms()`（F4 SDK 自带）替代 `delay_ms()`（F10x 自定义）

**4. Bug 修复**
- uvprojx 路径：`..\..\Firmware\` → `..\FIRMWARE\` / `..\CMSIS\`（28 处）
- SDK `_it.c` 中 `led_spark()` 调用过滤（SDK 示例代码，我们的模板不包含）

### 第五阶段：重构（2026-05-19 晚）

**1. 消除芯片硬编码**
- 启动文件过滤：`"startup_gd32f10x"` → 通用 `"startup_"` 前缀 + `_find_dir` 精确查找
- 设备头文件替换：`"gd32f10x.h" / "gd32f4xx.h"` 硬编码 → 用 chip JSON 的 `device_header` 字段
- `define` / `flash_driver`：取消家族级回退，改为每个芯片显式配置

**2. SDK 路径简化**
- 从 "每个系列单独配置路径" → "单一 SDK Root 目录"
- `resolve_sdk()`：按 `sdk_subdir` 前缀自动匹配目录名
- 找不着对应包给出明确错误提示

**3. 调试串口配置化**
- `retarget_printf.c` 中 `USART2` 写死 → `{{DEBUG_USART}}` 占位符，默认 `USART0`
- UART 模板同步使用占位符

**4. FWLIB 文件列表动态化**
- FWLIB 文件列表从 uvprojx 模板硬编码 → 运行时扫描 FIRMWARE 目录动态生成

### 第六阶段：STM32 系列（2026-05-19 晚 ~ 2026-05-20）

**1. STM32F10x 支持**
- 9 款芯片：C8T6 ~ ZET6，适配 STM32F10x_StdPeriph_Lib SDK 结构
- SDK 结构与 GD32 不同：CMSIS 在 `Libraries/CMSIS/`，固件库在 `Libraries/STM32F10x_StdPeriph_Driver/`
- `common_skip` 机制：跳过 `retarget_printf.c`（使用 GD32 API，不兼容 STM32）
- uvprojx 模板适配 STM32F1 pack 和 DLL 参数

**2. STM32F4xx 支持**
- 6 款芯片：F407×4 + F429×2，适配 STM32F4xx_DSP_StdPeriph_Lib SDK 结构
- Cortex-M4 + FPU2，core_cm4.h / core_cmSimd.h 头文件
- F4 GPIO API 与 F1 不同，模板需用 `GPIO_Mode_OUT` + `GPIO_OType_PP` + `GPIO_PuPd`（非 `GPIO_Mode_Out_PP`）
- UART TX 用 `GPIO_Mode_AF` + `GPIO_OType_PP`（非 `GPIO_Mode_AF_PP`），RX 用 `GPIO_Mode_AF` + `GPIO_PuPd_NOPULL`

**3. 占位符扩展**
- `{{CONF_HEADER}}`：conf/libopt 头文件占位符，消除 `stm32f10x_conf.h` / `gd32f10x_libopt.h` 硬编码
- `{{DEVICE_HEADER}}`：设备头文件占位符

**4. fwlib_exclude 机制**
- 芯片 JSON 可配置 `fwlib_exclude` 列表，排除不兼容的固件源文件
- 例如 STM32F407 排除 `stm32f4xx_fmc.c/h`（FMC 控制器不适用于该型号）

**5. Bug 修复**
- STM32F4 GPIO 时钟源：APB2 → AHB1 总线
- STM32F4 main.c：`stm32f10x_conf.h` → `stm32f4xx_conf.h`
- F4 core_cmSimd.h 文件名：驼峰大小写修正
- STM32 uvprojx：移除 retarget_printf 引用、修复 system 源文件路径、去掉不存在的 main.h
- `--diag_suppress=47`：消除 ST 头文件自带宏重定义警告

**6. 中英文语言切换**
- GUI 支持中文/英文界面切换

**7. README 更新（2026-05-20）**
- 反映当前实际支持的 4 系列 32 型号，补充 SDK 准备、fwlib_exclude 等新功能说明

## 关键设计原则

1. **数据驱动架构** — 加新芯片只需新增 JSON + 模板，不改 Python
2. **SDK 文件实时拷贝** — 工具不打包固件库，生成时从官方 SDK 目录获取最新版
3. **芯片专属文件从 SDK Template 取** — `*_it.c`、`systick.c/h`、`main.h` 等跟着 SDK 版本走
4. **单一 SDK Root** — 所有芯片固件包放同一目录，工具自动匹配
5. **每芯片独立配置** — `define`、`flash_driver`、`startup`、`device`、`fwlib_exclude` 都在 JSON 中显式声明
6. **占位符 > 硬编码** — 所有芯片差异通过 `{{PLACEHOLDER}}` 由模板引擎填充
