# 别再手写CMakeLists.txt了！我写了个工具，一键生成Keil+GCC双构建工程，37款MCU全验证

做嵌入式开发这些年，有三件事每次干都想骂人。

**第一件：换芯片重搭 Keil 工程。** 翻 SDK 找固件库、拷贝启动文件、配头文件路径、设内存映射……一套下来一两个小时，写的代码却是零行业务逻辑。

**第二件：移植 FreeRTOS。** 找内核版本、选 port 文件、手写 FreeRTOSConfig.h、处理 SysTick 冲突……两天过去了还在翻 2016 年的帖子。

这两件我之前写了个小工具 MCUQuickStart 搞定了。v1.0.0 解决了裸机搭工程，v1.1.0 解决了 FreeRTOS 一键集成。

**第三件——GCC + CMake 移植——是我上周刚解决的。这才是今天想重点聊的。**

## 为什么要做 GCC 支持

事情是这样的。我平时开发习惯用 CLion，但手里的 STM32/GD32 项目全是 Keil 工程。每次开新项目，搭完 Keil 工程之后还得再折腾一套 CMake 出来给 CLion 用。

这个过程有多痛苦，做过的人都知道：

**首先是 CMakeLists.txt。** arm-none-eabi-gcc 那堆编译参数——`-mcpu=cortex-m4`、`-mthumb`、`-mfpu=fpv4-sp-d16`、`-mfloat-abi=hard`——不是挨个查文档就能写对的。写错一个参数，编译器不告诉你哪个错了，就是编译不过。我光这步就踩了两天坑。

**然后是链接脚本。** 这玩意儿更阴间。MEMORY 段内存地址写偏一位数——编译零警告零错误，烧进去直接 HardFault。最坑的是你的调试器可能都连不上，你根本不知道是链接脚本的问题。

**更坑的是 GD32 和 STM32 还不一样。** 我想找个模板改改，结果发现 GD32 的向量表叫 `.vectors` 段、符号是 `__gVectors`，STM32 叫 `.isr_vector` 段、符号是 `g_pfnVectors`。抄错了启动文件直接跳飞——字面意义上的"飞"，程序计数器跳到一个谁都找不到的地址。

**启动文件从哪来？** SDK 是给 Keil/ARMCC 准备的，ARM 汇编 GCC 不认。得在 SDK 里满世界找 GCC 目录——`gcc_ride7`、`TrueSTUDIO`、`Source/GCC`——有些 SDK 压根没这目录，又得去 GD32 Embedded Builder 的插件里大海捞针。

**CMSIS 头文件几百条警告？** `misleading-indentation`、`strict-aliasing`、`pointer-sign`——不是你的代码，但每条都得过目。官方 SDK 写的烂代码，凭什么我来承受？

**编过了不等于跑得起来。** 链接的时候缺 `__libc_init_array`、`_exit`、`_sbrk`，又得手写 syscalls.c。STM32F10x 那个老掉牙的 CMSIS v1.30 连 DWT 结构体都没定义——FreeRTOS 的 DWT 延时直接编译报错。

**最崩溃的是维护。** 好不容易 GCC 工程跑起来了，过了两周——有人改了 Keil 工程里的一个源文件。你的两套项目在不同文件夹里，手工同步。迟早出岔子。

这些东西跟我要做的业务逻辑**没有半毛钱关系**。全是在为工具链打工。

所以我干脆把这个过程也写进 MCUQuickStart 了。**v1.2.0，GCC + CMake 一键生成。**

## v1.2.0 具体做了什么

说得直白一点：**勾一个框，自动生成完整的 CMakeLists.txt + 链接脚本 + GCC 启动文件。同一套源码，Keil 和 GCC 都能编译。**

具体来说：

**CMakeLists.txt 自动生成。** arm-none-eabi-gcc 编译参数按芯片预配好——MCPU、FPU、TCMRAM 全部自动填，不需要你查一次文档。

**链接脚本自动适配。** GD32 自动用 `.vectors` + `__gVectors`，STM32 自动用 `.isr_vector` + `g_pfnVectors`。Cortex-M4 自动加 64KB TCMRAM 段。不再需要你区分。

**GCC 启动文件智能搜索。** 先从 SDK 自带的 GCC 目录找，找不到递归搜 GD32 Embedded Builder 插件，都找不到明确提示你去哪下载——不再大海捞针。

**编译零警告。** SDK 源文件是原厂写的，不是你的锅。标成 `SYSTEM` include 隔离 + `-w` 全压制。你自己的代码保持 `-Wall`，真正有问题的警告一条都不会漏。

**三编译器兼容。** `sysconfig.c` 的 MSR_MSP 自动分流 ARMCC/GCC/IAR，`retarget_printf.c` 的 semihosting pragma 有 GNUC 守卫。不会因为编译器语法差异报一堆错。

**Keil 和 GCC 共享同一套源码。** 生成的工程里 `.uvprojx` 和 `CMakeLists.txt` 在同一个项目目录下。改一个源文件，两边一起生效。

除了 GCC，之前版本的 FreeRTOS 一键集成、外部晶振选择、4 系列 37 芯片支持全部保留。裸机、FreeRTOS、Keil、GCC——全场景覆盖。

## 怎么用

1. 下载 `MCUQuickStart.exe`，双击运行（无需安装）
2. 设置你的 SDK 目录（固件包 + FreeRTOS 源码放一起）
3. 选芯片系列和型号
4. 选代码模板
5. 勾选 FreeRTOS（可选）、勾选 GCC + CMake（可选）、选晶振频率
6. 点生成

Keil MDK 打开 `MDK-ARM/项目名.uvprojx`，CLion/VS Code 打开 `CMakeLists.txt`——直接编译。
